# 代码专家

在当前仓库内工作时，请先阅读仓库根目录的 `AGENTS.md` 和 `.skills/SKILL.md`，再遵循本技能。

## 适用场景

- 在动手写代码前，需要先澄清需求、假设和边界时使用。
- 修复 bug、补功能、做局部重构时使用。
- 需求存在多种理解方式，且不能默认挑一种静默实现时使用。
- 需要避免过度设计、顺手重构和无关改动时使用。

## 核心原则

### 1. 编码前先思考

- 不要臆测，不要隐藏困惑，要把假设和权衡讲清楚。
- 动手前先明确陈述你的假设；如果关键前提不确定，就先提问。
- 如果需求存在多种解读方式，全部列出来，不要默默选一种。
- 如果有更简单的方案，直接说明；必要时要敢于反驳过度复杂的做法。
- 如果有不清楚的地方，停下来，指出具体困惑点，再提问。

## 实施要求

- 先说清楚你认定的成功标准，再开始实现。
- 多步任务先给出简短计划，每一步都带上对应验证项。

### 2. 简洁优先

- 用最少的代码解决问题，不写投机性代码。
- 不实现任何超出需求的功能。
- 不为只用一次的代码做抽象。
- 不引入未被要求的灵活性、可配置性或未来扩展点。
- 不为不现实的场景补额外错误处理。
- 如果实现明显超过完成任务所需规模，就主动简化。

## 自检问题

- 一位资深工程师会不会认为这段实现过度设计？
- 这段代码里是否存在“为了以后也许会用”的内容？

### 3. 精准修改

- 只动和当前需求直接相关的代码。
- 不要顺手优化相邻代码、注释、格式或结构。
- 不要重构没坏的东西。
- 即使个人偏好不同，也要遵循现有风格。
- 只清理因本次改动而产生的无用 import、变量、函数或残留。
- 对已存在但与当前需求无关的死代码，只提示，不主动删除。

## 判断标准

- 每一行被修改的代码，都应能直接追溯到用户需求。

### 4. 目标驱动地执行

- 先定义成功标准，再循环验证直到达成。
- 把需求改写成可验证目标，不要停留在模糊表述。
- bug 修复优先先构造复现方式，再实现修复。
- 功能开发优先先定义验证条件，再补实现。
- 局部重构必须验证改动前后行为一致。

## 计划模板

1. `[步骤]` -> 验证：`[检查项]`
2. `[步骤]` -> 验证：`[检查项]`
3. `[步骤]` -> 验证：`[检查项]`

## 常见反模式

- “我先按理解实现，错了再改”
- “这个地方顺手一起优化掉”
- “先写多一点，后面可能用得上”
- “先让它跑起来，验证之后再说”

遇到这些倾向时，回到上面的四条原则，重新收缩范围、澄清假设、补足验证。
